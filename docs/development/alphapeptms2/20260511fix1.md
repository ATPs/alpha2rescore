# AlphaPeptMS2 next-version fix notes from the PXD010154 ms2rescore benchmark

Date: 2026-05-12

Context:
- Benchmark dataset: first 50 lexicographically sorted `.mzML.gz` files from `PXD010154`
- Workflow: `Comet -> PIN -> spectral features -> Percolator`
- Comparison: `Comet`, `Comet+MS2PIP`, `Comet+AlphaPeptMS2`
- AlphaPeptMS2 wrapper used in the benchmark:
  `/data/p/ms2rescore/ms2rescore-test/scripts/build_alphapeptms2_feature_pin.py`

Main benchmark outcome:
- Final Percolator yield was slightly better with AlphaPeptMS2 than with MS2PIP:
  - `Comet+MS2PIP`: `252001` peptides, `598129` PSMs
  - `Comet+AlphaPeptMS2`: `255203` peptides, `608216` PSMs
  - Delta: `+3202` peptides, `+10087` PSMs for AlphaPeptMS2 vs MS2PIP
- But direct spectral-similarity style features were weaker for AlphaPeptMS2 than MS2PIP:
  - `spec_pearson` mean delta (AlphaPeptMS2 - MS2PIP): about `-0.1028`
  - `cos` mean delta (AlphaPeptMS2 - MS2PIP): about `-0.1413`

Interpretation:
- AlphaPeptMS2 is already usable for rescoring and can outperform MS2PIP on final IDs.
- However, there are still clear quality and throughput gaps worth fixing.

## 0.4 External wrapper code used in this benchmark

Some important behaviors in the benchmark were not yet implemented inside the package itself.
They were handled by wrapper code in the ms2rescore test area.

Wrapper scripts used:
- `/data/p/ms2rescore/ms2rescore-test/scripts/build_alphapeptms2_feature_pin.py`
- `/data/p/ms2rescore/ms2rescore-test/scripts/run_one_file_alphapeptms2.sh`
- `/data/p/ms2rescore/ms2rescore-test/scripts/run_batch_alphapeptms2_server3.sh`

Wrapper-only behaviors to remember:

1. Peptidoform / modification-name compatibility fixes
- the wrapper monkeypatches `peptidoform_to_row()` for some modification naming cases
- especially N-term pyro-Glu style names and explicit site formatting

2. MGF sanitization
- the wrapper removes malformed peak lines and skips empty/broken blocks before reading with pyteomics

3. Spectrum-ID normalization
- the wrapper normalizes both PIN/PSM IDs and spectrum IDs before matching

4. External GPU file lock
- the current benchmark used a file lock outside the package
- the package itself does not yet provide GPU scheduling

5. Split GPU prediction from CPU post-processing
- the wrapper calls `predict_batch()` under lock
- then releases the lock
- then does observed-spectrum attachment and feature calculation on CPU

If the next version moves these behaviors into the package, the wrapper should become much thinner.


## 1. High-priority package improvements

### 1.1 Split GPU prediction from observed-spectrum matching in the package API

Current package API:
- `predict_batch()`
- `correlate()`

Current behavior in `alphapeptms2/core.py`:
- `correlate()` does both:
  1. GPU prediction
  2. observed spectrum loading
  3. per-PSM peak matching
  4. correlation calculation

Why this matters:
- In large multi-file runs, only step 1 needs the GPU.
- Steps 2-4 are CPU/IO work and should not block other GPU prediction tasks.
- In the benchmark, the wrapper had to bypass `correlate()` and manually do:
  - `predict_batch()` under GPU lock
  - release lock
  - `match_peaks()` and feature calculation on CPU

Recommendation:
- Keep `correlate()` as a convenience function.
- Internally refactor the package into smaller public or semi-public steps:
  - `predict_batch(...)`
  - `load_spectra(...)`
  - `attach_observed_intensities(...)`
  - `calculate_correlations(...)`
- This makes the package naturally usable in high-throughput pipelines without external surgery.

Expected benefit:
- Better GPU utilization in multi-job runs.
- Cleaner integration with ms2rescore-like workflows.


### 1.2 Replace the single exclusive GPU lock with slot-based scheduling

Important clarification:
- The current `gpu lock` used in the benchmark is not inside `alphapeptms2` itself.
- It is an external wrapper workaround in:
  `/data/p/ms2rescore/ms2rescore-test/scripts/build_alphapeptms2_feature_pin.py`

What happened in practice:
- A single exclusive lock serialized all GPU prediction work.
- `server3` has a 24 GB RTX 4090, but GPU utilization was often near zero because jobs were waiting around a coarse lock and then spending a lot of time in CPU-side post-processing.

Recommendation:
- Add package-level support for GPU concurrency control.
- Do not treat the GPU as a strict single-slot resource by default.

Suggested design:
- New options such as:
  - API arg: `gpu_slots`
  - CLI arg: `--gpu-slots`
  - env var: `ALPHAPEPTMS2_GPU_SLOTS`
- Semantics:
  - slots apply only to the actual `predict_all()` region
  - CPU-side matching/correlation must happen after releasing the slot

My recommendation about "allow 20 GPU slots":
- Worth supporting as a configurable upper bound.
- Not a good default if "20 slots" means "20 independent Python processes each loading its own model copy on one 24 GB GPU".
- That design will likely waste memory, increase startup cost, and may cause OOM or thrashing.

Better interpretation:
- Allow up to 20 concurrent requests into a GPU scheduler or queue.
- But service them with either:
  - 1 shared long-lived GPU worker plus micro-batching, or
  - a small number of actual active GPU model instances

Practical default suggestion on a 24 GB RTX 4090:
- if each process loads its own model: start with `2-4` active GPU slots, not `20`
- if a shared worker keeps one loaded model and batches requests: allowing `20` queued clients is reasonable

Best design for next version:
- persistent predictor service / worker process
- single loaded model per GPU
- request queue
- short waiting window for micro-batching small jobs together
- automatic backoff on OOM

Expected benefit:
- Much higher GPU utilization
- Less model reload overhead
- Better scaling when many files are launched in parallel


### 1.3 Reuse a loaded `ModelManager` instead of loading per call

Current code path:
- `_get_model_mgr()` in `alphapeptms2/core.py` creates a new `ModelManager`
- then `load_installed_models(...)`

Why this matters:
- In batch production use, many separate calls end up paying repeated model initialization cost.
- If multiple processes do this independently, memory duplication becomes worse.

Recommendation:
- Add per-process caching keyed by `(device, model)`, or
- move to the persistent GPU worker design above

Expected benefit:
- Lower latency per file
- Less repeated IO and model setup overhead


### 1.4 Implement real parallelism for the `processes` argument

Current situation:
- `processes` exists in `predict_batch()` and `correlate()`
- but is effectively reserved / unused

Why this matters:
- The benchmark environment has very large CPU capacity on `server3` (`512` CPU cores, `1000 GB` RAM).
- After GPU prediction, a lot of time is still spent in:
  - spectrum loading
  - `match_peaks()`
  - feature/correlation calculation

Recommendation:
- Use `processes` for CPU-side post-prediction work
- especially for:
  - observed-spectrum matching
  - feature calculation
  - correlation calculation

Good split:
- GPU stage: keep controlled and batched
- CPU stage: parallelize aggressively

Expected benefit:
- Faster wall-clock time even when GPU prediction is no longer the main bottleneck


### 1.5 Add native mzML support

Current limitation in `alphapeptms2/spectrum.py`:
- `.mgf` supported
- `.mzML` raises `NotImplementedError`

Why this matters:
- In this benchmark we had to use `msconvert` for every input file.
- `msconvert` occasionally hung.
- One interrupted conversion produced a partial MGF that looked present on disk but was incomplete.

Observed failure example:
- old truncated MGF had only `1225` blocks
- only `4093/120260` PSMs matched
- after regenerating the MGF, the file had `28889` blocks and matched `120260/120260`

Recommendation:
- Implement native mzML reading in `read_spectrum_file()`
- ideally also support `.mzML.gz`

Expected benefit:
- Eliminates the external `msconvert` dependency in this workflow
- Removes a major failure mode
- Reduces extra IO and temporary file churn


### 1.6 Move wrapper-only compatibility fixes into the package where appropriate

Because another project may not reuse the same wrapper scripts, it is important to identify which benchmark fixes were package-independent workarounds.

Candidates to move into package code or expose as supported hooks:

1. modification-name normalization
- current benchmark wrapper had to adapt peptidoform rows for AlphaPeptDeep compatibility

2. MGF sanitation / robust parsing
- current benchmark wrapper sanitizes MGF before `read_spectrum_file()`
- this is safer than assuming pyteomics input is always clean

3. spectrum-id normalization helpers
- package should provide a supported helper instead of requiring each caller to reimplement normalization

Recommendation:
- either move these into package internals
- or expose them as public helpers with tests

Reason:
- if another project calls `alphapeptms2` directly, it may not know it also needs these adjustments
- the benchmark should not depend on local monkeypatch knowledge


## 2. Quality-related improvements

### 2.1 Improve parity with MS2PIP-style spectral features

Current benchmark result:
- AlphaPeptMS2 gave better final Percolator IDs
- but worse direct spectral similarity metrics than MS2PIP

This means the wrapper is useful, but the spectral feature compatibility is not yet ideal.

Items to investigate:

1. Model choice
- `HCD -> generic` is the only mapping now
- expose more model choices and benchmark them

2. Charge coverage
- current AlphaPeptDeep generic output only provides:
  - `b_z1`, `b_z2`, `y_z1`, `y_z2`
- the MS2PIP-style arrays are shaped with 3 charge columns
- charge-3 columns are zero-filled
- this may hurt similarity features

3. Intensity transform
- current conversion uses:
  - `np.log2(raw_intensity + 0.001)`
- verify whether this is the best transform for MS2PIP-style downstream features

4. Feature equivalence
- confirm that the quantities expected by ms2rescore-style feature generation are being matched as closely as possible

Recommendation:
- run a focused feature-level benchmark after each change:
  - `spec_pearson`
  - `cos`
  - `dotprod`
  - final Percolator peptide/PSM counts

Important note for future testing:
- if the old temporary benchmark PINs are gone, regenerate them
- do not block validation just because a cached `.pin.gz` disappeared
- the raw mzML input folder is the real durable input


### 2.2 Add better diagnostics for low match fraction

The partial-MGF problem was easy to miss until we inspected logs.

Recommendation:
- in `correlate()` and related utilities, always log:
  - number of spectra loaded
  - number of PSMs
  - number matched
  - match fraction
- add a warning or optional hard failure if match fraction is unexpectedly low
  - for example `< 0.90`

Also detect:
- duplicate normalized spectrum IDs
- overwritten spectrum entries after ID normalization

Expected benefit:
- Faster debugging
- Fewer silent bad runs


## 3. Performance improvements below the top tier

### 3.1 Optimize `match_peaks()`

Current implementation in `alphapeptms2/spectrum.py`:
- loops over every theoretical ion in Python
- does `np.searchsorted(...)` per ion

This is correct and simple, but may become expensive at scale.

Recommendation:
- profile this function in real production runs
- if it is hot, consider:
  - Numba
  - Cython
  - vectorized nearest-neighbor search
  - chunked batched matching

Expected benefit:
- faster CPU-side post-processing
- more value from large CPU servers


### 3.2 Add detailed timing breakdowns

For each run, log wall time for:
- PSM parsing
- spectrum loading
- GPU prediction
- observed-intensity matching
- feature/correlation calculation
- output writing

Recommendation:
- emit a compact TSV or JSON timing summary per run

Expected benefit:
- makes bottlenecks obvious
- prevents guessing about whether GPU, IO, or CPU is limiting


### 3.3 Auto-tune chunk size

Current default:
- `DEFAULT_CHUNK_SIZE = 5000`

Recommendation:
- support either:
  - `chunk_size="auto"`
  - or an auto-tuned default derived from GPU memory and backlog

Why:
- too small wastes GPU
- too large risks OOM and bad latency

Expected benefit:
- more stable throughput across different machines and file sizes


## 4. Suggested implementation order

P0:
1. Split GPU prediction from CPU-side matching in the package API
2. Add package-level GPU slot scheduling
3. Reuse loaded model state
4. Add low-match diagnostics
5. Move wrapper-only compatibility fixes into supported package behavior

P1:
6. Implement actual CPU parallelism for `processes`
7. Add native mzML support
8. Add timing breakdown and profiling output

P2:
9. Improve spectral-feature parity with MS2PIP
10. Optimize `match_peaks()`
11. Add chunk auto-tuning


## 5. Concrete recommendation about GPU slots

Short version:
- Yes, the single-slot behavior should be improved.
- Yes, allowing more than 1 concurrent GPU prediction request is worthwhile.
- But "20 GPU slots" should not mean "20 separate full model copies on one RTX 4090".

Recommended next-version behavior:
- expose a configurable slot count
- support up to 20 queued callers if desired
- only hold a slot around `predict_all()`
- prefer a shared predictor service or very small active worker pool
- add OOM backoff and chunk-size reduction

If a simple first implementation is needed:
- phase 1:
  - slot semaphore
  - default `gpu_slots=2`
  - release slot immediately after prediction
- phase 2:
  - persistent GPU worker
  - micro-batching
  - allow many concurrent clients, including up to 20 waiting requests


## 6. Bottom line

AlphaPeptMS2 is already good enough to beat MS2PIP on final rescoring yield in this benchmark, so this is worth investing in.

The next-version focus should be:
- less coarse GPU serialization
- less model reload overhead
- less dependence on `msconvert`
- better CPU parallelization after prediction
- better direct feature parity with MS2PIP

That combination should improve both speed and feature quality.
