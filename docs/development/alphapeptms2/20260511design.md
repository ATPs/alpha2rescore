# alphapeptms2 — Design Document

**Date**: 2026-05-11
**Status**: Draft (pre-implementation)
**Goal**: Wrap AlphaPeptDeep behind an MS2PIP-aligned API so alphapeptms2 can serve as a drop-in
MS2 predictor for MS2Rescore (and other tools that already consume MS2PIP).

---

## 1. Motivation

- **MS2PIP** is the most widely used Python MS2 prediction tool with a clean API (`ms2pip.predict_single`,
  `ms2pip.predict_batch`, `ms2pip.correlate`). MS2Rescore integrates with it directly through the
  `MS2PIPFeatureGenerator` class in `ms2rescore/feature_generators/ms2pip.py`.
- **AlphaPeptDeep** (`peptdeep`) is a deep-learning alternative that predicts MS2, RT, and ion mobility.
  Its internal API (`ModelManager.predict_all`) uses pandas DataFrames with separate
  `sequence`, `mods`, `mod_sites`, `charge` columns and returns wide-format fragment DataFrames.
  This API is different enough from MS2PIP's that it cannot be used as a drop-in replacement.
- **alphapeptms2** adds a thin wrapper that imports `peptdeep` and exposes the same function
  signatures, input conventions, and output types as MS2PIP. This allows MS2Rescore (and similar
  tools) to call AlphaPeptDeep with minimal configuration changes — typically just swapping the
  feature generator import.

### 1.1 Non-goals (for this phase)

- Full MS2Rescore integration (separate future work — requires a new feature generator in MS2Rescore).
- `predict_library` / `get_training_data` / `annotate-spectra` commands from MS2PIP. We start with the
  core three: `predict_single`, `predict_batch`, `correlate`.
- Producing observed-intensity-based features identical to MS2PIP's. We aim for the same data
  structures and the same feature-calculation logic as MS2Rescore's current `MS2PIPFeatureGenerator`,
  but the predicted intensities themselves will differ because the underlying model is different.

## 3. API Design (Python)

All public functions are importable from `alphapeptms2` (i.e. in `__init__.py`).

### 3.1 `predict_single`

```python
def predict_single(
    peptidoform: Union[str, Peptidoform],
    model: str = "HCD",
    device: str = "cpu",
) -> ProcessingResult:
```

- **Input**: A ProForma string (e.g. `"PGAQANPYSR/3"` or `"RNVIM[Oxidation]DKVAK/2"`)
  or a `psm_utils.Peptidoform` instance.
- **Model**: AlphaPeptDeep model family name. Valid values are those supported by
  `ModelManager.load_installed_models()`. We default to `"HCD"` to match MS2PIP's default.
- **Processing steps**:
  1. Parse ProForma → `sequence`, `mods`, `mod_sites`, `charge` (see section 3.4).
  2. Initialize `ModelManager(device=device)`, load `model`.
  3. Call `model_mgr.predict_all(df, predict_items=["ms2"], frag_types=...)`.
  4. Convert the returned `fragment_mz_df` / `fragment_intensity_df` into the
     `ProcessingResult.theoretical_mz` and `ProcessingResult.predicted_intensity` dicts
     (see section 3.5).

### 3.2 `predict_batch`

```python
def predict_batch(
    psms: Union[PSMList, str, Path],
    model: str = "HCD",
    device: str = "cpu",
    psm_filetype: Optional[str] = None,
    processes: int = 1,
) -> List[ProcessingResult]:
```

- **Input**: A `psm_utils.PSMList` or a path to a PSM file in any format supported by `psm_utils`.
  If a file path is given, `psm_filetype` can override the auto-detected format.
- **Processing steps**:
  1. If `psms` is a file path: `psm_list = read_psms(psms, filetype=psm_filetype)`
     (reuse `ms2pip._utils.psm_input.read_psms` or write a minimal equivalent that calls
     `psm_utils.io`).
  2. Convert each PSM's `peptidoform` → AlphaPeptDeep DataFrame.
  3. Run `ModelManager.predict_all` on the combined DataFrame in chunks (batch size guided by
     `ModelManager` defaults, tunable via keyword).
  4. Slice results back to per-PSM `ProcessingResult` objects using the fragment-start/stop
     indices returned by AlphaPeptDeep.

### 3.3 `correlate`

```python
def correlate(
    psms: Union[PSMList, str, Path],
    spectrum_file: Union[str, Path],
    model: str = "HCD",
    device: str = "cpu",
    ms2_tolerance: float = 0.02,
    psm_filetype: Optional[str] = None,
    spectrum_id_pattern: Optional[str] = None,
    processes: int = 1,
) -> List[ProcessingResult]:
```

- **Input**: Same PSM input as `predict_batch` plus a path to an MGF or mzML spectrum file.
- **Processing steps**:
  1. Parse PSMs (as in `predict_batch`).
  2. Run AlphaPeptDeep prediction (as in `predict_batch`).
  3. Read the spectrum file and yield `ObservedSpectrum` objects (see section 4).
  4. Match each PSM to its observed spectrum by `spectrum_id` (with optional regex pattern).
  5. For each matched pair, annotate the observed peaks with theoretical fragment m/z values
     using the given `ms2_tolerance`, producing `observed_intensity` per ion type.
  6. Return `ProcessingResult` objects with both `predicted_intensity` and `observed_intensity`
     populated.

### 3.4 ProForma → AlphaPeptDeep conversion

AlphaPeptDeep expects a DataFrame with columns: `sequence`, `mods`, `mod_sites`, `charge`.

- `sequence`: stripped amino-acid string (e.g. `"PEPTIDEK"`).
- `mods`: semicolon-separated modification names (e.g. `"Oxidation"`), matching AlphaPeptDeep's
  convention.
- `mod_sites`: semicolon-separated 1-indexed positions (e.g. `"3"` for modification at position 3).
- `charge`: integer precursor charge.

ProForma parsing leverages `psm_utils.Peptidoform` (already available in alphabase env, v1.5.2):
- `Peptidoform.parsed_sequence` → `sequence`
- `Peptidoform.parsed_sequence[].modifications` → `mods` and `mod_sites`
- `Peptidoform.precursor_charge` → `charge`

Modification names must be resolvable by AlphaPeptDeep's built-in modification mapping
(`peptdeep.spec_lib.translate.mod_to_unimod_dict`). We need a fallback that maps common ProForma
mod names (like `"Oxidation"`) to the format AlphaPeptDeep expects. The existing
`peptdeep.settings.add_user_defined_modifications` mechanism handles this.

### 3.5 Fragment type mapping (AlphaPeptDeep → MS2PIP)

AlphaPeptDeep currently predicts these fragment types (confirmed 2026-05-11 from the `generic` model):

| AlphaPeptDeep column  | MS2PIP ion_type | MS2PIP charge index |
|-----------------------|-----------------|---------------------|
| `b_z1`                | `"b"`           | 0                   |
| `b_z2`                | `"b"`           | 1                   |
| *(not predicted)*     | `"b"`           | 2 (z3 — zero-fill)  |
| `y_z1`                | `"y"`           | 0                   |
| `y_z2`                | `"y"`           | 1                   |
| *(not predicted)*     | `"y"`           | 2 (z3 — zero-fill)  |
| `b_modloss_z1`        | `"B"`           | 0 (future)          |
| `b_modloss_z2`        | `"B"`           | 1 (future)          |
| `y_modloss_z1`        | `"Y"`           | 0 (future)          |
| `y_modloss_z2`        | `"Y"`           | 1 (future)          |

**ProcessingResult dictionary layout** (aligns to MS2PIP):

```python
result.theoretical_mz = {
    "b": np.array([[b1_mz, b2_mz, 0.0],  # per fragment position
                   [b1_mz, b2_mz, 0.0],
                   ...]),  # shape (n_frag_positions, 3)
    "y": np.array([[y1_mz, y2_mz, 0.0],
                   ...]),
}
result.predicted_intensity = {
    "b": np.array([[b1_int, b2_int, 0.0], ...]),  # log2 space
    "y": np.array([[y1_int, y2_int, 0.0], ...]),
}
```

AlphaPeptDeep returns wide DataFrames where row i is a fragment position, columns are
`b_z1, b_z2, y_z1, y_z2`. We reshape:
1. Extract columns → (n_rows, 2) for b, (n_rows, 2) for y.
2. Pad with zeros for z3.
3. Apply `np.log2(x + 0.001)` to intensities (AlphaPeptDeep outputs raw intensities; MS2PIP stores
   log2 internally).

### 3.6 `ProcessingResult` definition

Mirrors `ms2pip.result.ProcessingResult`:

```python
class ProcessingResult(BaseModel):
    psm_index: int
    psm: Optional[PSM] = None
    theoretical_mz: Optional[Dict[str, np.ndarray]] = None
    predicted_intensity: Optional[Dict[str, np.ndarray]] = None
    observed_intensity: Optional[Dict[str, np.ndarray]] = None
    correlation: Optional[float] = None
    model_config = ConfigDict(arbitrary_types_allowed=True)
```

## 4. Spectrum Handling (for `correlate`)

### 4.1 Reading spectrum files

We need a lightweight MGF/mzML reader. MS2PIP uses `ms2rescore-rs` (Rust) for this, but we cannot
depend on that in the alphabase environment. Options:

**Preferred approach**: Use `pyteomics` (already available as a dependency of `psm_utils` in
alphabase env, v1.5.2) to read MGF files, and `pymzml` or a minimal mzML parser for mzML files.
If mzML support adds too many dependencies, start with MGF-only and add mzML later.

Example MGF reader signature:

```python
def read_spectrum_file(spectrum_file: Union[str, Path]) -> Generator[ObservedSpectrum, None, None]:
    """Yield ObservedSpectrum objects from an MGF or mzML file."""
```

`ObservedSpectrum` is a simple data class (or pydantic model) holding:
- `mz: np.ndarray` (float32)
- `intensity: np.ndarray` (float32)
- `identifier: str` (spectrum title / nativeID)
- `precursor_mz: float`
- `precursor_charge: int`
- `retention_time: Optional[float]`

### 4.2 Peak matching (lightweight)

For `correlate`, after prediction we need to extract observed intensities at each theoretical
fragment m/z. This is a simple nearest-match within tolerance:

```
for each theoretical_mz in fragment ions:
    find observed peaks where |mz_obs - mz_theo| < ms2_tolerance
    take the intensity of the closest match (or 0 if none)
```

MS2PIP uses Cython for this (`ms2pip_pyx.get_targets`). We implement a pure numpy version
for portability, since the Python overhead is negligible for single-spectrum matching.

## 5. CLI Design

Entry point `alphapeptms2` registered via pyproject.toml. Subcommands follow MS2PIP conventions:

### 5.1 `alphapeptms2 predict-single`

```bash
alphapeptms2 predict-single "PGAQANPYSR/3" --model HCD --device cpu
alphapeptms2 predict-single "RNVIM[Oxidation]DKVAK/2" --model HCD --device gpu
```

Options:
- `--model`: model name (default: "HCD", matching MS2PIP)
- `--device`: "cpu" or "gpu" (default: "cpu")
- `--output`: optional path to write result as CSV/TSV

### 5.2 `alphapeptms2 predict-batch`

```bash
alphapeptms2 predict-batch peptides.tsv --model HCD --device gpu --output predictions.csv
```

Options:
- `--model`, `--device`: as above
- `--output`: path for predictions CSV
- `--psm-filetype`: override file format detection
- `--processes`: number of workers for multiprocessing (default: 1)

### 5.3 `alphapeptms2 correlate`

```bash
alphapeptms2 correlate peptides.tsv spectra.mgf --model HCD --ms2-tolerance 0.02
```

Options:
- `--model`, `--device`, `--psm-filetype`, `--processes`: as above
- `--ms2-tolerance`: Da tolerance for peak matching (default: 0.02)
- `--spectrum-id-pattern`: regex for extracting spectrum ID from spectrum titles
- `--output`: path for results CSV

## 9. Open Questions / Risks

1. **Modification name compatibility**: ProForma strings may contain modification names
   (like `"Oxidation"`, `"Carbamidomethyl"`) that AlphaPeptDeep recognizes through its
   `mod_to_unimod_dict`. We need to verify that common unimod names resolve correctly.
   If not, we need a name-mapping layer.

2. **Peptide length limits**: AlphaPeptDeep's current model is trained for peptides up to ~50
   amino acids. MS2PIP supports up to 100. Longer peptides will raise an error; this should
   be documented.

3. **Model file location**: AlphaPeptDeep downloads pretrained models to
   `/data/users/x/peptdeep/pretrained_models/` by default. This path is controlled by the
   `PEPTDEEP_HOME` setting. The wrapper must set this correctly before calling `ModelManager`.

4. **z3 (charge 3 fragment) ions**: AlphaPeptDeep does not currently predict z3 fragment ions.
   We zero-fill these positions in the `ProcessingResult` arrays. This is acceptable because
   MS2Rescore's feature calculator can handle zeros, and the correlation features for z3 ions
   will simply be 0. If this becomes a limitation, a future AlphaPeptDeep model may add z3
   prediction.

5. **modloss ions**: AlphaPeptDeep supports `b_modloss_z1`, `b_modloss_z2`, `y_modloss_z1`,
   `y_modloss_z2`, but MS2PIP uses a different convention for neutral-loss peaks (they are
   separate ion types in some models). For the initial version, we focus on non-modloss b/y
   ions only, matching MS2PIP's HCD model ion types.

6. **Multiprocessing**: AlphaPeptDeep has its own multiprocessing logic in
   `ModelManager.predict_all(multiprocessing=True, ...)`. We need to test interaction with
   any multiprocessing we add for `predict_batch`. For initial release, we can set
   `multiprocessing=False` in AlphaPeptDeep and handle parallelism ourselves if needed.

7. **Intensity scale**: AlphaPeptDeep outputs raw (linear) intensities. MS2PIP stores log2
   internally (`log2(x + 0.001)`). MS2Rescore's feature calculator expects log2 values and
   converts them back to linear with `2**x - 0.001`. We must ensure round-trip consistency.
